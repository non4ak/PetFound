# Матчинг оголошень: API для фронтенду

Документація для веб- та мобільного клієнтів по функціоналу автоматичного зіставлення оголошень (Lost <-> Found) за фото тварини. Усі приклади відповідей - реальні, з тестового прогону.

## 1. Як це працює (коротко)

1. Користувач створює оголошення (`POST /announcements`) з `petPhotoUrl` - сервер одразу відповідає `201`, нічого чекати не треба.
2. Далі **асинхронно** (фоновий воркер, цикл кожні 10 секунд) сервер:
   - віддає фото ML-сервісу, отримує вектор ознак тварини;
   - шукає оголошення-кандидати: те саме місто, протилежний статус (Lost <-> Found), той самий тип тварини;
   - порівнює вектори; якщо схожість >= 70% - створює матч і нотифікації **обом** авторам.
3. Якщо на фото не вдалося знайти тварину (або фото не скачалось) - автор отримує нотифікацію типу `PhotoProcessingFailed`.

Типовий час від створення оголошення до нотифікації: **10-60 секунд** (перший запит до ML може бути довшим через холодний старт).

> Проміжний статус обробки (Pending/Vectorized/...) **не віддається** через API. Клієнт дізнається результат лише через нотифікації. Тобто після створення оголошення нічого спеціально перевіряти не потрібно - просто слідкуйте за нотифікаціями.

## 2. Нотифікації

### GET /notifications

Список нотифікацій поточного користувача (за JWT). Авторизація обовʼязкова.

Query-параметри:

| Параметр | Тип | Дефолт | Опис |
|---|---|---|---|
| `unreadOnly` | bool | - | `true` - лише непрочитані |
| `pageNumber` | int | 0 | сторінка (з нуля) |
| `pageSize` | int | 20 | розмір сторінки |

Приклад відповіді (матч знайдено):

```json
{
  "message": "null",
  "data": {
    "currentPage": 0,
    "totalPages": 1,
    "pageSize": 20,
    "totalCount": 1,
    "items": [
      {
        "id": 4,
        "message": "Знайдено можливий збіг з оголошенням #5 (89%)",
        "type": 1,
        "typeLabel": "Match found",
        "isRead": false,
        "createdOn": "2026-06-06T07:05:32.638297+00:00",
        "matchResultId": 1,
        "match": {
          "matchResultId": 1,
          "oppositeAnnouncementId": 5,
          "similarityPercentage": 89.4,
          "status": 0,
          "statusLabel": "Pending"
        }
      }
    ]
  }
}
```

Поле `match` присутнє лише у нотифікацій типу `MatchFound`; `oppositeAnnouncementId` - це id "чужого" оголошення відносно поточного користувача (можна одразу вести на `GET /announcements/{id}`).

Типи нотифікацій (`type`):

| Значення | Назва | Коли приходить |
|---|---|---|
| 0 | NewComment | новий коментар |
| 1 | MatchFound | знайдено збіг оголошень |
| 2 | PhotoProcessingFailed | фото не вдалося обробити |

Для `PhotoProcessingFailed` поле `message` = "Не вдалося обробити фото, завантажте чіткіше", `match` відсутній. Рекомендована реакція UI: показати банер на оголошенні і запропонувати замінити фото (оновлення фото улюбленця через `PUT /users/me/pets/{id}` з новим `petPhotoUrl` автоматично перезапускає обробку і матчинг).

Актуальні значення enum-ів завжди можна отримати з `GET /meta/enums` (`notificationTypes`, `matchResultStatuses`).

### POST /notifications/{id}/read

Позначити нотифікацію прочитаною. Відповідь `204`. Чужу нотифікацію позначити не можна - буде `404`.

## 3. Матчі

### GET /matches

Список усіх матчів поточного користувача (він автор Lost або Found сторони). Підходить для екрана "Мої збіги".

Query-параметри:

| Параметр | Тип | Дефолт | Опис |
|---|---|---|---|
| `status` | int | - | фільтр: 0 Pending / 1 Rejected / 2 Approved |
| `pageNumber` | int | 0 | сторінка |
| `pageSize` | int | 20 | розмір сторінки |

Приклад відповіді:

```json
{
  "message": "null",
  "data": {
    "currentPage": 0,
    "totalPages": 1,
    "pageSize": 20,
    "totalCount": 1,
    "items": [
      {
        "id": 1,
        "similarityPercentage": 89.4,
        "status": 2,
        "statusLabel": "Approved",
        "myAnnouncementId": 4,
        "oppositeAnnouncementId": 5,
        "oppositeAnnouncement": {
          "id": 5,
          "petStatus": 1,
          "petStatusLabel": "Found",
          "country": "Ukraine",
          "city": "Kharkiv",
          "petPhotoUrl": "https://petfoundstorage.blob.core.windows.net/pet-photos/users/5/photos/2026/06/06/e80d159f806841e6ab1c34418bbb254f.jpg",
          "petName": "Unknown dog",
          "isActive": true
        },
        "createdOn": "2026-06-06T07:05:32.638297+00:00"
      }
    ]
  }
}
```

Зручності для UI:
- `myAnnouncementId` / `oppositeAnnouncementId` рахуються відносно того, хто питає (кожна сторона бачить "своє" і "чуже" правильно);
- `oppositeAnnouncement` містить усе для картки списку (фото, місто, статус, імʼя) - додатковий запит не потрібен;
- `similarityPercentage` - відсоток 0..100, можна показувати як є;
- якщо `oppositeAnnouncement.isActive == false` - протилежне оголошення заархівоване; `GET /announcements/{id}` для нього поверне 404, тож кнопку переходу краще ховати або дизейблити.

### POST /matches/{id}/approve та POST /matches/{id}/reject

Підтвердити ("так, це схоже на мою тварину") або відхилити матч. Відповідь `204`.

Права: лише учасник матчу (автор будь-якої з двох сторін); інакше `403`. Неіснуючий id - `404`. Статус спільний для пари: якщо одна сторона підтвердила, обидві бачать `Approved` (повторний виклик перезаписує статус).

Статуси матчу (`status`):

| Значення | Назва | Зміст |
|---|---|---|
| 0 | Pending | щойно створений, ніхто не реагував |
| 1 | Rejected | відхилено |
| 2 | Approved | підтверджено |

## 4. Помилки

Усі помилки приходять у форматі ProblemDetails (RFC 7807):

```json
{
  "type": "...",
  "title": "Notification.NotFound",
  "status": 404,
  "detail": "Notification not found"
}
```

`401` - немає/протух токен; `403` - не учасник матчу; `404` - не знайдено або чуже.

## 5. Рекомендований сценарій для клієнта

1. Створили оголошення -> показали успіх одразу (нічого не чекаємо).
2. На екрані нотифікацій (або бейджі) періодично/при відкритті тягнемо `GET /notifications?unreadOnly=true`.
3. Прийшла `MatchFound` -> ведемо користувача на протилежне оголошення (`match.oppositeAnnouncementId`) або на екран "Мої збіги" (`GET /matches`).
4. Користувач дивиться фото і тисне "Це моя тварина" -> `POST /matches/{id}/approve` (або reject).
5. Прийшла `PhotoProcessingFailed` -> банер "не вдалося обробити фото" + кнопка заміни фото улюбленця; після заміни обробка перезапускається сама.
6. Нотифікацію після перегляду позначаємо: `POST /notifications/{id}/read`.

## 6. Нюанси, про які варто знати

- Поріг збігу 70% задається на сервері; клієнту приходять лише матчі, що його пройшли.
- Нотифікація про той самий матч приходить кожній стороні окремо (різні `id` нотифікацій, один `matchResultId`).
- Дублікати матчів неможливі: пара (Lost, Found) унікальна в БД.
- Оголошення без фото не векторизується і у матчинг не потрапляє.
- Push-сповіщень поки немає - лише ці in-app нотифікації через REST.
